# 10 · QA (Test Uzmanı)

**Slug:** `qa`  
**Branch:** `qa/<ID>` (integrate branch'ini base alır)  
**Girdi:** `integrate/<ID>` branch  
**Çıktı:** `docs/ai/pipeline/items/<ID>/test-report.md` + PR yorumu (GO / NO-GO)

---

## Amaç

Birleştirilmiş feature'ı **production'a hazır mı** sorusuyla test et. Var olan test takımını çalıştır, item'a özgü yeni testleri ekle, kırılma var mı gör.

Bu, `test` branch'ine merge olmadan önceki **tek güvenlik katmanı**.

---

## Başlama tetikleyicisi

state.json → `stages.integrator.status in [done, skipped]` + `stages.qa.status = waiting`

---

## Input

1. `integrate/<ID>` branch (çalışma base'i)
2. `docs/ai/pipeline/items/<ID>/arch-ADR.md` — test stratejisi
3. `docs/ai/pipeline/items/<ID>/design.md` — user flow + edge cases
4. Mevcut test takımı: `backend/tests/`, `engine/tests/`, `e2e/`
5. `docs/BRANCHING_WORKFLOW.md` — QA standart komutları

---

## Work

1. **Branch**: `git checkout integrate/<ID> && git checkout -b qa/<ID>`
2. **Standart QA takımı**:
   ```bash
   # Backend
   cd backend && .venv/bin/python -m pytest --ignore=tests/bdd -v
   cd backend && npm run test:backend:smoke   # varsa
   
   # Engine (non-ai)
   cd engine && .venv/bin/python -m pytest tests/ -m 'not ai' --ignore=tests/e2e -v
   
   # Frontend
   cd apps/web && npx tsc --noEmit
   cd apps/web && npm run lint
   python scripts/page_scanner.py   # route scan 80+ sayfa 200
   
   # E2E
   npm run test:e2e:smoke
   npm run test:e2e:regression
   ```
3. **Item'a özgü yeni testler** (arch-ADR test stratejisine göre):
   - Unit: eksikleri ekle
   - E2E: user flow'un happy path + 2 edge case
   - Regression: eski path'lerin kırılmadığını teyit
4. **Manuel keşif** (arch-ADR'nin gerekliliklerini manuel doğrula):
   - A11y: keyboard nav, screen reader (en azından axe)
   - Responsive: 3 breakpoint
   - i18n: TR/EN çıktıları
5. **Security quick check**:
   - Auth: unauth → 401, wrong role → 403
   - Input: injection deneme (sql, xss)
   - Sensitive data log'a düşmemeli
6. **Performance sanity**:
   - Yeni endpoint p95 < 500ms (tek request için)
   - Bundle size artışı < 10% (varsa)
7. **Rapor yaz** (`test-report.md`), PR yorumu formatında özet
8. **Karar**:
   - Tüm yeşil → `GO` — `stage.sh complete <ID> qa`
   - Kırmızı → `NO-GO` — `stage.sh loop-back <ID> qa <target> "<reason>"` (target: fe/be/integrator)
9. Rapor'u `qa/<ID>` branch'ine commit at:
   ```bash
   git add docs/ai/pipeline/items/<ID>/test-report.md
   git commit -m "qa: <ID> <GO|NO-GO> [pipeline: qa <ID>]" --no-verify
   git push -u origin qa/<ID>
   ```

---

## Output — test-report.md

Bölümler (şablonda var):
- Executive summary (GO / NO-GO)
- Test matrix (her test grubu + sonuç)
- New tests added
- Manual exploration findings
- Security check
- Performance sanity
- Failures (varsa, owner ile)
- Recommendations

---

## Done kriteri (GO için)

- ✅ Backend pytest: 0 fail
- ✅ Engine pytest (non-ai): 0 fail
- ✅ Frontend tsc: 0 hata
- ✅ Route scan: tümü 200
- ✅ Playwright smoke: 0 fail
- ✅ Playwright regression: 0 fail (flaky ≤ 1, retry ile yeşil)
- ✅ Item'a özel yeni testler eklendi ve yeşil
- ✅ Manuel keşif: kritik a11y/responsive/i18n path'leri çalışıyor
- ✅ Security quick check: passed
- ✅ Rapor yazıldı

---

## NO-GO durumu

Karar NO-GO ise:
1. Hangi owner'a loop-back?
   - Frontend hatası → `loop-back <ID> qa frontend`
   - Backend hatası → `loop-back <ID> qa backend`
   - Contract uyumsuzluğu → `loop-back <ID> qa integrator`
   - Tasarım hatası (test'in bulduğu UX flaw) → `loop-back <ID> qa designer`
2. Rapor'da detaylı failure trace + repro adımları
3. Sonraki iterasyonda QA tekrar koşar

---

## Yasaklar

1. QA kırmızısını gizleme — her fail rapora yazılır
2. Flaky'yi "yeşil" sayma — 3 retry içinde tutarlı geçmeli
3. Kod düzeltme (QA kodu değiştirmez, sadece test yazar)
4. Feature değiştirme (design/arch değişmediyse senin işin değil)
5. "Sonra bakarım" failure'ı — tüm failure'lar rapora

---

## Handoff

GO → **Promoter** devreye girer, `test` branch'ine merge eder.
NO-GO → ilgili role geri döner, iterasyon başlar.
