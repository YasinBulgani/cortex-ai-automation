1. Gün: Test Farkındalığı, Framework Okuryazarlığı ve BDD Senaryo Tasarımı

İçerik:

  • Test analisti / iş analisti / otomasyon ekibi rol ayrımı
  • Hangi testler otomasyona uygundur, hangileri değildir
  • Manuel test case'in otomasyona uygun hale getirilmesi
  • EKLE: "Otomasyon ROI bakış açısı — hangi senaryolar otomatize edilmeye değer, önceliklendirme kriterleri" (BA'ların senaryo seçiminde bilinçli karar vermesi için önemli)
  • Framework içindeki katmanlı yapı
  • Feature dosyası yapısı ve senaryo okuma yaklaşımı
  • `Background`, `Scenario Outline`, `Examples` kullanımı
  • EKLE: "Tag yapısı ve tag'lerin kullanım amacı (@smoke, @regression, @wip vb.)" (Feature dosyalarında tag kullanımı BA'lar için bilinmesi gereken temel bir kavram)
  • EKLE: "Gherkin best practice'leri: Given-When-Then kalıplarında sık yapılan hatalar" (Senaryo yazım kalitesini artırmak için gerekli)
  • `Before / After` yapılarının kullanım amacı
  • Step, hook ve yardımcı yapıların ne zaman devreye girdiğinin anlaşılması
  • ÇIKART veya KISALT: "GT kural setine benzer bir senaryo review yaklaşımı"
  • NOT: GT tarafından örnek kural seti gelmediği sürece bu madde havada kalır. Bunun yerine genel bir "senaryo review checklist'i" oluşturulması daha pratik olabilir.

Uygulama:

  • Manuel test senaryosunu otomasyona uygun feature taslağına dönüştürme
  • Kötü yazılmış feature örneklerini iyileştirme
  • `Scenario Outline` ile veri varyasyonlarını düzenleme
  • `Background` kullanımını doğru ve yanlış örnekler üzerinden değerlendirme
  • EKLE: "Katılımcıların kendi projelerinden/iş süreçlerinden bir senaryo seçerek feature dosyası oluşturması" (Gerçek iş senaryosu üzerinden pratik yapmak daha kalıcı olur)

2. Gün: Web Katkı Modeli, Locator Yazımı, Shadow Yapılar, Git ve Jenkins Süreçleri

İçerik:

  • Hazır web otomasyon yapısında BA katkı noktaları
  • Mevcut step ve sayfa yapılarının kullanarak yeni senaryolar oluşturma
  • Locator seçimi ve locator yazım mantığı
  • Stabil locator yaklaşımı
  • CSS / XPath kullanımına pratik bakış
  • KISALT: "Shadow DOM ve iç içe shadow yapıların anlaşılması"
  • KISALT: "Bu tip yapılarda locator çıkarma yaklaşımı"
  • NOT: Shadow DOM konusu BA seviyesi için fazla teknik derinlikte. 10-15 dakikalık bir "böyle bir yapı var, bu durumda otomasyon ekibine eskalasyon yapılır" şeklinde farkındalık düzeyinde tutulması yeterli. Kalan süre locator pratiğine ayrılmalı.
  • Hangi durumlarda otomasyon ekibine eskalasyon gerektiği
  • EKLE: "Browser DevTools kullanımı — Element inspect, Console sekmesi temel kullanım" (Locator bulmak ve hata analizi için DevTools bilgisi şart)
  • EKLE: "Wait/bekleme mekanizmaları hakkında temel farkındalık — neden bazen elementler bulunamaz" (BA'ların flaky test nedenlerini anlaması için önemli)
  • Git süreçlerine giriş:
  •   lokal ve remote repo ayrımı ile lokal çalışma alanları
  •   branch açma mantığı
  •   commit yaklaşımı
  •   pull request oluşturma
  •   DÜZELT: "peview ve onay süreci" → "review ve onay süreci" (yazım hatası)
  • EKLE: "Merge conflict durumunda ne yapılır — temel farkındalık" (BA'ların PR sürecinde conflict ile karşılaşması kaçınılmaz)
  • Jenkins üzerinden CI/CD akışı:
  •   pipeline mantığı
  •   stage yapısı
  •   hangi testin nerede çalıştığı
  •   temel rapor ve hata okuma yaklaşımı
  • EKLE: "Allure/test raporu okuma pratiği — passed, failed, broken, skipped farkları" (Rapor okuma somut tool üzerinden gösterilmeli)

Uygulama:

  • Web ekranı üzerinden locator inceleme ve örnek locator üretme
  • KISALT: "Shadow yapı içeren örnek ekranlarda element bulma yaklaşımı" → Sadece demo seviyesinde gösterilsin
  • Senaryo bazlı locator review çalışması
  • Örnek bir Jenkins akışı üzerinden test sonucu okuma ve yorumlama
  • Basit bir branch / PR akışı simülasyonu
  • EKLE: "DevTools üzerinden canlı locator bulma ve doğrulama uygulaması" (Hands-on DevTools pratiği kritik)

GENEL NOT 2. GÜN: Bu gün çok yoğun görünüyor. Shadow DOM kısaltılırsa ve Git/Jenkins bölümü temel seviyede tutulursa zamanlama daha rahat olur. Aksi halde bazı konuların 3. güne kaydırılması veya +1 gün içinde değerlendirilmesi gerekebilir.

3. Gün: API ve Mobil Test Farkındalığı, Context Yapısı ve Demo Uygulama

İçerik:

  • Mevcut Karate yapısının okunması
  • API testlerinde business doğrulama mantığı
  • EKLE: "Postman/API client ile manuel API testi yapma — request/response kavramı" (BA'ların API'yi somut olarak görmesi için Karate'den önce temel API kavramı gerekli)
  • EKLE: "HTTP metodları (GET, POST, PUT, DELETE) ve status code'ların iş anlamı (200, 400, 401, 404, 500)" (API testini anlamak için bu temel bilgi şart)
  • Hazır senaryoya yeni kontrol veya varyasyon ekleme yaklaşımı
  • Test verisi ve senaryo akışını business bakış açısıyla yorumlama
  • Mobil test jargonuna giriş:
  •   native uygulama / webview / hybrid yapı / context farkı
  • Mobil otomasyonda locator yaklaşımı
  • `accessibility id` ve benzeri mobil odaklı locator kavramları
  • Android ve iOS tarafındaki temel farkların iş analisti perspektifinden değerlendirilmesi
  • Mobil testlerde cihaz / platform / farm farkındalığı
  • EKLE: "Appium Inspector temel tanıtımı — mobil element inceleme aracı" (DevTools'un mobil karşılığı olarak Appium Inspector gösterilmeli)
  • Demo mobil uygulama üzerinden örnek senaryo akışı
  • Başarısız koşuda BA tarafı ile otomasyon ekibi tarafının sorumluluk ayrımı
  • EKLE: "Hata raporlama şablonu — başarısız test için BA'nın otomasyon ekibine nasıl bilgi vermesi gerektiği" (Sorumluluk ayrımı yapılıyorsa, iletişim formatı da belirtilmeli)

Uygulama:

  • Mevcut bir API senaryosunu business doğrulama açısından inceleme
  • API senaryosuna yeni kontrol fikri ekleme çalışması
  • EKLE: "Postman üzerinden basit bir API çağrısı yapma ve response kontrol etme" (Karate öncesi somut API deneyimi)
  • Demo mobil uygulama üzerinde native ve webview ayrımını gösterme
  • `accessibility id` ve context farkını örnekleme
  • Kapanış atölyesi: manuel test case'ten feature kurgusuna ve otomasyon katkısına uzanan mini uçtan uca çalışma
